# 风控与监控体系设计

> 版本：v2.0(对齐三态状态机 + sidecar 推理 + 全 TS 实现;熔断阈值与分层逻辑沿用 v1.0)
> 基础文档：data-sources.md、decision-engine-design.md、implementation-plan-v1.md
> 定位：保障完全自动运行的 Agent 的安全性，覆盖熔断、告警、PnL 异常诊断
> 核心前提：用户不响应任何通知，所有风控必须自动执行
> 实施次序:风控骨架(risk_events 表、checkPreTick、EXTREME 全撤路径)在 W1 先于模型落地——
> 对托管产品,熔断与审计日志是产品本体,模型是可换的 alpha

---

## 一、设计前提与原则

### 1.1 关键前提

Agent **完全自动运行**，假设用户不会查看通知或介入。这意味着：

- 所有风控措施必须**自动触发、自动执行**，不能等人工确认
- 告警是给运维（开发方）看的，不是给用户的退出建议
- 任何"等待人工决策"的设计都是失败的设计

### 1.2 风控分层

```
第一层：预防（决策时）—— 在决策阶段就避免危险操作
第二层：熔断（实时）—— 危险发生时立即止损
第三层：监控（持续）—— 发现异常并告警
第四层：诊断（事后）—— 分析异常根因，改进系统
```

### 1.3 风控优先级

```
资金安全 > 止损 > 收益
```

任何时候，保护本金优先于追求收益。宁可错过机会，不可遭受不可逆损失。

## 二、风险分类

### 2.1 风险全景

| 风险类别 | 具体风险 | 主要来源 |
|---|---|---|
| 市场风险 | 库存暴露、趋势套牢、闪崩 | 价格波动 |
| 协议风险 | Cetus 漏洞、Lending 协议异常 | 智能合约 |
| 执行风险 | 交易失败、状态不一致、RPC 故障 | 链上执行 |
| 模型风险 | 预测失效、分布漂移 | ML 模型 |
| 数据风险 | 数据源故障、数据污染 | 数据采集 |
| 操作风险 | 私钥安全、配置错误 | 系统运维 |

### 2.2 各风险的防控手段映射

| 风险 | 预防 | 熔断 | 监控 |
|---|---|---|---|
| 库存暴露 | 库存修正、单侧偏置 | 失衡度熔断 | 库存失衡度监控 |
| 趋势套牢 | TREND 状态撤退 | drift 熔断 | drift_strength 监控 |
| 闪崩 | 预测前瞻 | 价格波动熔断 | 价格波动监控 |
| 协议漏洞 | — | TVL 异常熔断 | 池子健康监控 |
| 执行失败 | 提交前模拟 | — | 交易成功率监控 |
| 模型失效 | 降级到规则 | — | 预测准确率监控 |
| 数据故障 | 备用源 | 全源故障进 EXTREME | 数据新鲜度监控 |

## 三、第一层：预防性风控（决策时）

在决策阶段就避免危险，这是成本最低的风控。

### 3.1 库存暴露预防

```
决策时主动控制库存方向（来自 decision-engine）：
  失衡度 0.15-0.30 → 调整 ask/bid 权重
  失衡度 0.30-0.50 → 停止失衡方向新挂单
  失衡度 > 0.50 → 仅单侧挂单修正
```

### 3.2 成本基础保护

```
ask 底价规则：不挂低于成本+双fee+利润的 ask
防止价差变亏损
```

### 3.3 接管前评估

```
拒绝不适合的 PM（来自 decision-engine 第十节）：
  范围太窄、active 位置偏、资金太少、当前 EXTREME
从源头避免接管高风险 PM
```

### 3.4 资金分散

```
单一 lending 协议上限 60%（避免协议风险集中）
预防单点协议风险
```

## 四、第二层：熔断机制（实时止损）

### 4.1 熔断触发条件

任一条件触发，立即进入 EXTREME 状态（撤回所有挂单 + 全部 lending）：

| 触发条件 | 阈值 | 检测频率 | 风险类别 |
|---|---|---|---|
| 价格剧烈波动 | 5min 波动 > 10% | 每分钟 | 市场（闪崩）|
| 池子 TVL 骤降 | 5min 跌 > 50% | 每分钟 | 协议 |
| 跨市场价差异常 | Cetus vs Binance > 5% 持续 30s | 每 30s | 协议/数据 |
| 预测穿越概率 | p_break > 0.7 | 每次决策 | 市场（前瞻）|
| 累计亏损 | 24h PnL < -5% | 每分钟 | 综合 |
| 数据全源故障 | 所有交易所数据不可用 | 实时 | 数据 |
| Lending 协议异常 | 单协议 TVL 5min 跌 > 30% | 每分钟 | 协议 |

### 4.2 熔断的分级

不是所有熔断都"全撤"，按严重程度分级：

```
L1 软熔断（降低暴露）：
  触发：价差 0.5-1%、单一信号轻度异常
  动作：提高 lending 比例、收窄做市范围
  不全撤，降低风险敞口

L2 硬熔断（进入 EXTREME）：
  触发：上表的主要条件
  动作：撤回所有挂单、100% lending、1min 监控

L3 紧急停机（停止所有操作）：
  触发：私钥可能泄漏、合约检测到攻击、RPC 完全失联
  动作：停止一切链上操作，仅保持监控，等待运维介入
```

### 4.3 熔断执行的特殊性

```
熔断操作绕过常规 gas 预算筛选（避险优先于省 gas）
熔断 PTB 优先级最高，必要时提高 gas budget 确保成交
熔断后强制链上对账（极端行情后状态易乱）
```

### 4.4 熔断退出条件

```
EXTREME 退出需同时满足：
  - 所有触发条件解除
  - 持续稳定 10min（防止假性恢复）
  - 链上对账通过
退出后渐进恢复：先小仓位试探，确认正常再全量
```

### 4.5 熔断的防抖

```
避免在临界点反复熔断/恢复：
  熔断阈值与恢复阈值不同（滞回）
    例：> 10% 熔断，< 7% 才考虑恢复
  恢复需持续稳定时长
  短时间内多次熔断 → 延长恢复观察期
```

## 五、第三层：监控体系（持续）

### 5.1 监控分类

```
实时监控（秒级）：熔断相关指标
近实时监控（分钟级）：状态、持仓、PnL
低频监控（小时/天）：模型质量、策略表现
```

### 5.2 系统健康监控

| 监控项 | 指标 | 告警阈值 |
|---|---|---|
| 数据源 | 各源新鲜度、心跳 | 延迟 > 5s |
| 预测 sidecar | /health 探活、推理延迟、fallback 占比 | 探活失败、p99 > 2s、fallback 占比 > 20% |
| 决策循环 | 评估耗时、决策频率 | 超时、异常频率 |
| 链上执行 | 交易成功率、确认时间 | 成功率 < 95% |
| RPC 节点 | 可用性、延迟 | 不可用 |
| Agent 进程 | CPU、内存、存活 | 资源异常 |

### 5.3 持仓与资金监控

| 监控项 | 指标 | 告警阈值 |
|---|---|---|
| 库存失衡度 | sui_overage | > 0.5 持续 |
| 资金利用率 | 做市资金占比 | 异常偏离 |
| Lending 持仓 | 各协议分布 | 单协议 > 60% |
| 总资产 | NAV（净值） | 异常变动 |
| 老化库存 | 老化批次数量、占比 | 过多 |

### 5.4 策略行为监控

| 监控项 | 指标 | 异常含义 |
|---|---|---|
| 状态分布 | 各状态时间占比 | EXTREME 过多 = 市场恶劣或误判 |
| 状态切换频率 | 单位时间切换次数 | 过高 = 状态机抖动 |
| 调仓频率 | 单位时间调仓数 | 过高 = gas 浪费 |
| 预测使用率 | 用预测 vs 降级规则的比例 | 降级过多 = 预测服务问题 |

### 5.5 收益监控

| 监控项 | 指标 | 告警 |
|---|---|---|
| 实时 PnL | 当前净值 vs 期初 | 回撤超阈值 |
| 滚动收益 | 24h / 7d 收益率 | 持续为负 |
| 收益来源 | 价差/fee/lending 占比 | 结构异常 |
| 对比基线 | vs HODL、vs lending | 持续跑输 |

## 六、告警体系

### 6.1 告警分级

```
P0 紧急（立即处理，可能需运维介入）：
  - L3 紧急停机触发
  - 疑似私钥泄漏、合约攻击
  - 总资产异常大幅下跌
  - Agent 进程崩溃

P1 重要（尽快关注）：
  - L2 硬熔断频繁触发
  - 预测服务持续失败
  - 交易成功率持续低
  - 单日亏损超阈值

P2 注意（记录跟踪）：
  - L1 软熔断
  - 单次交易失败
  - 数据源短时延迟
  - 库存失衡度偏高

P3 信息（仅记录）：
  - 正常状态切换
  - 常规调仓
  - 模型重训完成
```

### 6.2 告警去重与聚合

```
避免告警风暴：
  相同告警短时间内聚合（如"RPC 失败 × 10"而非 10 条）
  持续性问题只在状态变化时告警（开始/结束）
  设置告警频率上限
```

### 6.3 告警通道

```
P0：多通道即时（电话/PagerDuty + IM + 邮件）
P1：IM + 邮件
P2：IM 或日志看板
P3：仅日志/看板
```

**注意**：所有告警面向运维（开发方），不向 PM 用户发送。用户侧 Agent 完全自动，不依赖用户响应。

## 七、第四层：PnL 异常诊断

### 7.1 诊断的目标

当 PnL 异常（大幅亏损、收益结构反常）时，自动分析根因，区分：

```
是市场原因（无法避免，趋势市本就难）
还是策略原因（参数不当、状态误判）
还是系统原因（执行失败、模型失效、数据污染）
```

### 7.2 PnL 分解诊断

利用归因体系（来自 decision-engine 第九节）逐层拆解：

```
总 PnL 异常 → 拆解到各来源：
  价差收益异常？
    → round-trip 完成率低？（库存积累，趋势套牢）
    → 价差/fee 比 < 1？（做市方向选择错误）
  Fee 收益异常？
    → 成交频率低？（范围设置不当、市场冷清）
  Lending 收益异常？
    → APR 下降？协议问题？
  Gas 消耗异常？
    → 调仓过频？（状态机抖动）
  未实现亏损大？
    → 库存暴露过高？（趋势市未及时撤退）
```

### 7.3 分状态诊断

```
按状态拆分 PnL，定位问题状态：
  哪个状态在持续亏钱？
  TREND 亏损 → 趋势市本就难，但检查反向仓位是否过大、偏置方向是否反了
  NORMAL 亏损 → 异常，检查预测质量、k_w/容忍偏移参数
  EXTREME 频繁 → 检查是否误判（熔断阈值太敏感）
```

### 7.4 归因到根因

```
诊断决策树示例：

24h 亏损 > 5%
├─ 未实现亏损主导？
│   └─ 是 → 库存暴露问题
│       ├─ 处于趋势市？→ 市场原因，检查是否该更早撤退
│       └─ 处于震荡市？→ 异常，预测中心可能偏差大
├─ 已实现价差为负？
│   └─ 是 → 做市方向错误
│       ├─ 预测准确率正常？→ 决策逻辑问题
│       └─ 预测准确率下降？→ 模型失效，检查漂移
├─ Gas 消耗异常高？
│   └─ 是 → 调仓过频，状态机抖动或阈值问题
└─ Lending 异常？
    └─ 是 → 协议问题，检查协议健康
```

### 7.5 自动诊断报告

```
异常触发时自动生成诊断报告（给运维）：
  - 异常指标快照
  - PnL 分解（来源、状态）
  - 时间线（异常前后的状态切换、决策、市场）
  - 初步根因判断
  - 相关日志链接
```

### 7.6 LLM 辅助诊断（未来扩展）

```
预留接口（呼应 prediction-service 的外部信号设计）：
  LLM agent 综合市场环境、新闻、链上数据生成事故分析
  解释"为什么这次亏损"，关联宏观事件
  这是 LLM 的优势区（跨模态信息整合 + 推理）
```

## 八、模型风险专项

### 8.1 模型失效检测

| 检测项 | 方法 | 响应 |
|---|---|---|
| 推理失败 | sidecar 超时/探活失败 | mlAgent 整体让位 Tier 0 策略(`emaTrend`),fallback 落库 |
| 预测漂移 | 滚动 pinball 上升 | 告警 + 触发重训评估 |
| 分布漂移 | 特征 PSI > 0.25 连续 3 个 1h 窗 | fallback="psi" → Tier 0;告警 + 重训 |
| 校准失效 | 在线区间覆盖率偏离离线 > 5pp | 告警;持续偏离 → Tier 0 |

### 8.2 模型与规则的安全切换

```
两态语义(可用 / 不可用,无连续混合——见 decision-engine-design.md §3.4)：
  fallback = false → 预测驱动决策
  fallback ≠ false → mlAgent 整体让位 Tier 0 规则策略,原因落 predictions 表
  恢复:连续 3 次成功推理且 PSI < 0.25
任何模型问题都不会导致 Agent 停摆——规则始终能兜底,且每次降级都留痕(无静默路径)
```

## 九、资金安全专项

### 9.1 私钥安全

```
Agent keypair 私钥安全存储（KMS/HSM/加密）
派生地址守卫：启动时验证地址正确
不可逆操作前二次校验
```

### 9.2 操作边界硬校验

```
每个链上操作提交前，代码级硬校验（不只靠模拟）：
  - 操作在 PM 范围内
  - 不操作 active bin
  - 不超出资金
  - 目标地址正确（防止资金误转）
这是最后一道防线，独立于决策逻辑
```

### 9.3 异常资金变动检测

```
监控 NAV 突变：
  非市场原因的资产减少 → P0 告警 + L3 停机
  防止漏洞/攻击导致的资金损失扩大
```

## 十、技术实现

### 10.1 模块划分(全 TS,挂在现有运行时内)

熔断检测的"实时"是分钟级轮询 + 事件触发,不是微秒级撮合——TS 完全够用,无需独立进程或 Rust。
操作边界硬校验在 PTB 构造层(`txUnified.ts`)做代码级断言,与执行层一体。

| 模块 | 位置 | 说明 |
|---|---|---|
| L1 软熔断 / L2 进 EXTREME | `src/risk/monitor.ts` / `circuits.ts` | rebalancer tick 前置 + 独立 1min 巡检 |
| L3 紧急停机 | `src/risk/emergency.ts` | 停止链上操作,仅监控 |
| 操作边界硬校验 | `txUnified.ts` 内断言 | PM 范围内、不碰 active bin、资金充足 |
| 指标采集 + 告警分发 | `src/risk/` + `src/lib/logger.ts` | 结构化日志 + risk_events 表 |
| PnL 归因诊断 | `src/risk/pnlAttribution.ts` | 消费 rebalances / predictions / market_state_history |
| 深度分析 / 看板 | `ml/`(Python,离线) | pandas 消费 SQLite + parquet |

### 10.2 监控数据存储(与仓库现状一致,不引入新存储)

```
风控事件 → SQLite risk_events 表(src/db/schema.sql)
预测与状态 → predictions / market_state_history 表
训练数据与特征行 → parquet(ml/data/),SQLite 不存第二份
决策日志 → 现有 rebalances 表 + 结构化日志
```

## 十二、关键设计决策回顾

1. **完全自动，不依赖用户**：所有风控自动执行，告警面向运维而非用户
2. **四层防御**：预防（决策时）→ 熔断（实时）→ 监控（持续）→ 诊断（事后）
3. **熔断分级**：软熔断（降暴露）、硬熔断（EXTREME）、紧急停机（停操作），不一刀切
4. **资金安全优先于收益**：宁可错过机会，不可不可逆损失
5. **模型问题不致停摆**：规则始终兜底，预测失效自动降级
6. **操作边界硬校验**：独立于决策逻辑的最后防线，代码级强制
7. **PnL 自动诊断**：区分市场/策略/系统原因，自动生成根因报告
8. **熔断防抖**：滞回 + 稳定时长，避免临界点反复横跳
