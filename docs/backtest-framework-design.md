# 回测框架设计

> 版本：v2.0(对齐 sidecar/三态方案;撮合引擎 Rust → Python 向量化)
> 基础文档：data-sources.md、prediction-service-design.md、decision-engine-design.md、implementation-plan-v1.md
> 核心难点：DLMM bin 级成交的真实模拟
> 技术栈：Python(撮合模拟 + 数据准备 + 分析,numpy 向量化)+ TS(策略逻辑复用,经 backtest/replay.ts)
>
> **两条铁律(v2.0 新增,源于 v1.0 计划评审)**:
> 1. **L1/L2 回测的绝对 PnL 不作为任何验收 gate**——fill 模型用 Binance 量代理 Cetus 链上流,
>    系统性偏差未知(Binance 量是 Cetus 的 10–100×)。L1 只做**同模拟器下的相对排序**
>    (mlAgent vs multiBinSpot vs 基线)。
> 2. **fee 模型必须用 Cetus 真实 swap 事件标定**(`ml/backtest/fee_model.py`,数据来自 W1 的
>    历史扫块),不用 Binance 量直接外推。可得历史不足时,在报告中标注标定窗口与置信区间。
>
> 模型本身的主验收 gate 是预测质量指标(pinball / 覆盖率 / 方向准确率,见
> prediction-service-design.md §3.3)——可证伪且与模拟器无关;回测负责策略层的相对比较与压力测试。

## 一、回测的根本挑战

### 1.1 为什么 DLMM 回测比普通策略难

普通 CEX 策略回测只需要 OHLCV，因为成交逻辑简单（市价单立即成交，限价单价格到了就成交）。但 DLMM 做市的回测有三个特殊难点：

**难点 1：成交依赖 bin 级别的价格穿越**

我们的盈亏不取决于"价格涨跌"，而取决于"价格是否穿过了我们挂单的 bin，穿了多少量"。需要模拟价格在离散 bin 间的移动，以及每次穿越带来的成交。

**难点 2：成交量分配依赖池子整体流动性**

每笔 swap 的 fee 和成交按"我们在该 bin 的流动性占比"分配。要准确模拟，需要知道**整个池子在每个 bin 的流动性**，而不只是我们自己的。

**难点 3：反身性——我们的挂单影响价格**

我们添加流动性会改变池子深度，理论上会影响价格冲击。小资金可忽略，大资金不能。

### 1.2 回测的分级目标

不追求一步到位的完美模拟，分三级：

| 级别 | 目标 | 数据需求 | 用途 |
|---|---|---|---|
| L1 简化回测 | 验证策略逻辑正确性 | Binance OHLCV | 快速迭代、参数粗调 |
| L2 中等保真 | 估算收益区间 | Binance OHLCV + 池子流动性快照 | 收益评估、风险评估 |
| L3 高保真 | 接近实盘 | Cetus 历史 bin 级成交数据 | 上线前最终验证 |

L1 先做（数据易得），L3 需要采集真实数据后才能做。

## 二、价格路径与成交模拟

### 2.1 价格数据的处理

**主数据源**：Binance SUIUSDC（流动性深、代表市场共识，见 data-sources）。

**问题**：Binance 是连续价格，Cetus 是离散 bin。需要把 Binance 价格映射到 Cetus bin 序列。

```
bin_id(price) = floor(log(price / base_price) / log(1 + bin_step))
  bin_step = 0.005 (0.5%)
```

### 2.2 价格路径重建

OHLCV 只有开高低收，缺少 bin 间移动的路径。需要重建价格如何穿越 bin：

**问题示例**：一根 5min K 线 open=$1.00, high=$1.02, low=$0.99, close=$1.01。价格是先到 high 还是先到 low？这影响成交顺序。

**重建策略**（按保真度递增）：

```
策略 A（保守，L1）：
  假设最坏路径——先到不利方向再到有利方向
  open → low → high → close（如果我们有 bid 在低位、ask 在高位）
  这样高估了对我们不利的成交，给出保守收益估计

策略 B（OHLC 顺序推断，L2）：
  用更细粒度数据（1min 替代 5min）减少单根 K 线内的不确定性
  1min 内假设线性穿越

策略 C（真实 tick，L3）：
  用 Binance 逐笔成交重建真实价格路径
  最接近真实穿越顺序
```

### 2.3 bin 穿越判定

```python
def bins_crossed(from_price: float, to_price: float) -> list[BinCross]:
    """给定一段价格移动,判定穿过了哪些 bin(成交量分配见 2.4)"""
    from_bin, to_bin = bin_id(from_price), bin_id(to_price)
    step = 1 if to_bin > from_bin else -1
    return [BinCross(bin_id=b, direction=step)
            for b in range(from_bin + step, to_bin + step, step)]
```

### 2.4 成交量分配模型（最关键）

价格穿过一个 bin 时，我们成交多少？这取决于**穿越该 bin 的总成交量**和**我们在该 bin 的流动性占比**。

```
我们在 bin k 的成交量 = 穿越 bin k 的总 swap 量 × (我们的流动性 / bin k 总流动性)
```

三个级别的处理：

**L1（简化）**：假设我们的挂单全部成交

```
价格穿过我们的 bin → 该 bin 我们的全部流动性成交
优点：简单
缺点：高估成交（实际可能只成交一部分），但对"价差是否盈利"的逻辑验证够用
```

**L2（按量比例）**：用 Binance 成交量代理池子成交量

```
该 bin 成交量 ≈ Binance 同期成交量 × (Cetus池子量 / Binance量) 的历史比例
我们的成交 = bin成交量 × 流动性占比
需要：池子在各 bin 的流动性分布（快照或估算）
```

**L3（真实数据）**：用 Cetus 历史 swap 事件

```
直接用采集的 Cetus 历史 bin 级成交数据
精确知道每个 bin 实际成交了多少、我们占多少
```

### 2.5 池子流动性建模

L2/L3 需要知道"别人"在各 bin 的流动性（决定我们的占比）。

```
L1：忽略（假设全成交）
L2：用池子 TVL + 假设的分布形状估算
    - 定期快照池子总 TVL
    - 假设流动性集中在 active bin 附近（实测分布或 spot 形状）
L3：用 Cetus 历史快照重建每个时刻的真实 bin 流动性
```

## 三、回测引擎架构

### 3.1 事件驱动设计

回测引擎是一个事件驱动的循环，复用实盘的决策逻辑：

```
┌─────────────────────────────────────────────┐
│           回测主循环（时间推进）              │
│                                             │
│  for each time_step in historical_data:      │
│    1. 推进市场状态（价格、成交量）            │
│    2. 模拟价格穿越 → 我们的挂单成交            │
│    3. 更新持仓、库存、PnL                     │
│    4. 若到决策时点：                          │
│       a. 构造市场快照                         │
│       b. 调用预测（用历史特征 → 模型推理）     │
│       c. 调用决策逻辑（复用实盘代码）          │
│       d. 模拟执行调仓（含 gas、滑点）          │
│    5. 记录状态                                │
└─────────────────────────────────────────────┘
```

### 3.2 关键原则：复用实盘代码

回测的策略逻辑必须与实盘**完全一致**，否则回测无意义。

```
实盘代码 ←─ 共享 ─→ 回测代码
  状态机判定（src/state/machine.ts）        ✓ 直接复用
  目标分布计算（mlAgent + binWeights.ts）    ✓ 直接复用
  库存修正（src/decision/inventory.ts）      ✓ 直接复用
  成本基础保护（src/decision/ageStopLoss.ts）✓ 直接复用
  差量计算（src/decision/diffPlanner.ts）    ✓ 直接复用
  预测推理（Python sidecar / 离线批量 predict)✓ 同一份 ml/features/ 代码

  数据来源：实盘=实时 API，回测=历史回放    ← 唯一区别
  执行：实盘=真实 PTB，回测=模拟撮合        ← 唯一区别
```

通过依赖注入隔离"数据源"和"执行器"，策略核心不变。

```typescript
interface DataFeed {
  getMarketSnapshot(time: Timestamp): MarketSnapshot;
}
interface Executor {
  execute(actions: BinDiff[]): ExecutionResult;
}
// 实盘：LiveDataFeed + ChainExecutor
// 回测：HistoricalDataFeed + SimulatedExecutor
```

### 3.3 预测模型在回测中的处理

**关键陷阱：前视偏差（look-ahead bias）**

回测时调用预测模型，必须保证模型只用 T 时刻之前的信息。

```
1. 模型必须用"截至回测时点之前"的数据训练
   不能用整个历史训练再回测全程（数据泄漏）
2. 用 walk-forward 验证：
   训练窗口 [0, T] → 回测 [T, T+Δ]
   滚动前进，每次只测试训练数据之后的时段
3. 特征计算严格用历史数据，标签绝不进入特征
```

### 3.4 walk-forward 回测流程

```
将历史数据切成滚动窗口：
  窗口1：训练[1-3月] → 测试[4月]
  窗口2：训练[1-4月] → 测试[5月]
  窗口3：训练[1-5月] → 测试[6月]
  ...
每个测试段用对应窗口训练的模型
拼接所有测试段 → 完整的样本外回测曲线
```

这模拟了实盘中"定期重训、用最新模型预测未来"的真实情况。

## 四、成本与摩擦建模

回测必须包含真实摩擦，否则收益虚高。

### 4.1 Gas 成本

```
每次调仓操作 gas ≈ 0.02 SUI × 当时价格
按 PTB 实际操作数计算（不是按 bin 数，PTB 打包多个操作）
EXTREME 等强制操作也计入 gas
```

### 4.2 滑点与成交价

```
DLMM 成交价 = bin 价格（离散，无传统滑点）
但 fee 要扣：每笔成交扣 0.4%
价格穿越多个 bin 时，按各 bin 价格分段成交
```

### 4.3 Lending 收益

```
闲置资金按 supply 时段的历史 APR 计息
APR 用历史真实值（采集 Scallop/Kai 历史利率）
切换 supply/redeem 的 gas 也计入成本
```

### 4.4 延迟与执行偏差

```
实盘：决策到链上确认有 1-3s 延迟，价格可能已变
回测建模：决策用 T 时刻价格，执行按 T+延迟 时刻价格
保守起见，延迟期内的不利价格变动算作执行偏差
```

## 五、PnL 归因（回测与实盘一致）

回测输出的 PnL 必须可分解，便于诊断收益来源。

```
总 PnL = 已实现价差 + 累计 fee + Lending 收益 - Gas - 未实现盈亏变化

已实现价差：FIFO 配对 bid/ask 成交，每完成一对计算
累计 fee：每笔成交累加
Lending 收益：supply→redeem 的资产增量
Gas：每笔 tx 实际消耗
未实现盈亏：当前库存按 mark price - 按 cost basis
```

### 5.1 分状态归因

```
按状态机状态拆分 PnL（NORMAL / TREND / EXTREME）：
  NORMAL 状态贡献了多少？
  TREND 状态是否在亏钱（趋势市本就难赚）？反向仓位与偏置方向是否正确？
帮助判断哪些状态的策略需要改进
```

### 5.2 分来源归因

```
价差 vs fee vs lending 各占多少？
  价差/fee 比值 < 1 → 做市方向选择有问题
  lending 占比过高 → 是否过于保守、错失做市机会
```

## 六、评估指标

### 6.1 收益指标

| 指标 | 说明 |
|---|---|
| 总收益率 | 期末净值 / 期初 - 1 |
| 年化收益率 | 按时间折算 |
| 价差收益占比 | 价差 / 总收益 |
| Fee 收益占比 | fee / 总收益 |
| Lending 收益占比 | lending / 总收益 |

### 6.2 风险指标

| 指标 | 说明 |
|---|---|
| 最大回撤 | 净值从峰值的最大跌幅 |
| 波动率 | 收益的标准差 |
| Sharpe 比率 | 风险调整后收益 |
| Sortino 比率 | 只惩罚下行波动 |
| 库存暴露分布 | 库存失衡度的时间分布 |
| VaR / CVaR | 尾部风险 |

### 6.3 做市质量指标

| 指标 | 健康范围 | 异常含义 |
|---|---|---|
| 价差/fee 比 | 1.5 - 3.0 | <1 方向选择有问题 |
| 平均 round-trip 时长 | < 2h | 长 = 库存积累 |
| Round-trip 完成率 | > 70% | <50% 大量库存未回归 |
| 资金利用率 | — | 做市资金 / 总资金 |
| 调仓频率 | — | 过高=gas 浪费，过低=反应慢 |

### 6.4 预测质量指标

| 指标 | 说明 |
|---|---|
| 中心 bin MAE | 预测中心 vs 实际 VWAP bin |
| 宽度校准 | 预测的 80% 区间实际覆盖率 |
| 预测贡献度 | 用预测 vs 不用预测（纯规则）的收益差 |

## 七、对照基线

回测必须有基线对比，否则不知道策略好在哪。

| 基线 | 说明 | 验证什么 |
|---|---|---|
| 纯持有（HODL） | 50/50 持有 SUI/USDC 不动 | 做市是否跑赢躺平 |
| 纯 lending | 全部资金吃 APR | 做市是否值得承担风险 |
| 对称做市（无预测） | 绕 active bin 对称挂单 | 预测是否真的有用 |
| 固定范围做市 | 不用状态机，固定 ±5 | 状态机是否有用 |
| 单一状态策略 | 永远 NORMAL 行为 | 自适应是否有用 |

**关键对比：有预测 vs 无预测**。这直接验证 prediction-service 的价值。如果带预测的回测收益不显著高于对称做市，说明预测模型没学到东西，需要回去改模型。

## 八、压力测试

除了历史回测，针对极端场景专门测试。

### 8.1 历史极端事件重放

```
2025年5月 Cetus 被攻击事件期间
2025年10月 SUI 闪崩
其他历史大波动时段
→ 验证 EXTREME 熔断是否及时、损失是否可控
```

### 8.2 合成极端场景

```
构造历史未出现但可能的场景：
  - 持续单边下跌 30%（验证库存暴露上限）
  - 闪崩后快速反弹（验证是否过早/过晚回归）
  - 长期窄幅横盘（验证 fee 收益是否覆盖 gas）
  - 流动性枯竭（验证 lending 是否安全）
```

### 8.3 参数敏感性分析

```
对关键参数做扫描，看收益对参数的敏感度：
  - k_w(halfWidth 系数)与容忍偏移推导
  - Gas 倍数要求
  - TREND 反向仓位比例与 trendBias 收缩阈值
  - 状态切换阈值(drift_strength、p_break)
过于敏感的参数 = 过拟合风险，需要稳健化
```

## 九、技术实现

### 9.1 模块划分

| 模块 | 语言 | 理由 |
|---|---|---|
| 撮合模拟(bin 穿越、成交分配) | Python(numpy 向量化) | 52 万行 1min 数据的数组运算,向量化后秒级 |
| 价格路径重建 | Python | 同上 |
| 策略逻辑 | TS(复用实盘,经 `src/backtest/replay.ts` / `cli.ts`) | 保证一致性 |
| 预测推理 | Python **批量** `booster.predict(X)` | 整矩阵一次推理,比逐行调用快几个数量级;与 sidecar 共享 ml/features/ |
| 数据准备、特征工程 | Python(复用训练管线) | 同一份代码 |
| 结果分析、可视化 | Python | pandas、matplotlib |

撮合是"对每个时间步做 bin 区间判定 + 按占比分配成交量"的数组运算,numpy 向量化即可分钟级跑完
全年数据;参数扫描用进程池并行。这条路径上没有 Rust 的立足点(推理侧结论见
prediction-service-design.md §1.2)。

### 9.2 性能要求

```
1 年 1min 数据 ≈ 52 万个时间点
单次完整回测:numpy 向量化 < 1 min
walk-forward(含逐折重训)< 1h;参数扫描数百次 → multiprocessing 并行
```

### 9.3 项目结构


## 十、回测可信度检查清单

回测结果可信的前提（避免自欺）：

```
□ 无前视偏差：特征只用历史，预测模型 walk-forward
□ 无数据泄漏：标签不进特征，训练测试严格时序隔离
□ 含全部摩擦：gas、fee、延迟、滑点都建模
□ 成交不高估：L1 的"全成交"假设要在 L2/L3 修正
□ 策略代码与实盘一致：复用而非重写
□ 有基线对比：至少对比 HODL、纯 lending、对称做市
□ 压力测试通过：极端场景损失可控
□ 参数不过拟合：敏感性分析稳健
□ 样本外验证：不在训练数据上评估
□ 多时段验证：牛市、熊市、震荡市分别测试
```

## 十二、关键设计决策回顾

1. **三级保真度**：L1 验证逻辑、L2 估算收益、L3 接近实盘，按数据可得性递进
2. **成交量分配是核心难点**：L1 假设全成交、L2 按量比例、L3 用真实数据
3. **复用实盘代码**：策略逻辑、预测推理直接复用，只换数据源和执行器
4. **walk-forward 防前视偏差**：滚动训练+测试，模拟真实的定期重训
5. **有预测 vs 无预测对照**：直接验证预测模型价值，是最关键的基线对比
